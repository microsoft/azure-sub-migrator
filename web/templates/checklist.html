Migration Checklist

Subscription: {{ subscription_id }}
Overall Progress 0 / 0 completed
0%
Phase 1: Pre-Transfer Actions 0
  • Export RBAC role assignments and custom roles
    All role assignments and custom roles are permanently deleted during transfer.
    Export all role assignments to JSON
    az role assignment list --subscription {{ subscription_id }} --all --output json > rbac_assignments.json
    Export custom role definitions
    az role definition list --custom-role-only true --subscription {{ subscription_id }} --output json > custom_roles.json
    Or use Tenova CLI
    tenova export-rbac -s {{ subscription_id }}
  • Map source principals to target tenant principals
    Create a principal mapping JSON file that maps old user/group/SP object IDs to their equivalents in the target tenant.
    List all unique principals in current assignments
    az role assignment list --subscription {{ subscription_id }} --all --query "[].principalId" -o tsv | sort -u
  • Back up Key Vault secrets, certificates, and keys
    Key Vault access policies are wiped during transfer. Export secrets and certificate references.
    List all Key Vaults in the subscription
    az keyvault list --subscription {{ subscription_id }} --query "[].name" -o tsv
    For each vault, export access policies
    # Replace <VAULT_NAME> with each vault name: az keyvault show --name "<VAULT_NAME>" --subscription {{ subscription_id }} \ --query "properties.accessPolicies" --output json > kv_policies.json
  • Export and remove resource locks
    Resource locks can block post-transfer remediation. Export before transfer.
    Export all resource locks
    az lock list --subscription {{ subscription_id }} --output json > resource_locks.json
  • Export Azure Policy assignments and custom definitions
    Policy assignments are permanently deleted during transfer.
    Export policy assignments
    az policy assignment list --subscription {{ subscription_id }} --output json > policy_assignments.json
    Export custom policy definitions
    az policy definition list --subscription {{ subscription_id }} \ --query "[?policyType=='Custom']" --output json > custom_policies.json
    Phase 2: Execute Transfer
    • Verify readiness check passes (no blockers)
      All blockers must be resolved before transfer.
      tenova readiness-check -s {{ subscription_id }}
    • Schedule maintenance window and notify stakeholders
      Subscription transfer requires downtime. Ensure all teams are aware.
    • Transfer the subscription to the target tenant
      Execute the directory change via Portal or CLI.
      Azure Portal (open in browser)
      https://portal.azure.com/#@/resource/subscriptions/{{ subscription_id }}/changeDirectory
      Tenova CLI
      tenova transfer -s {{ subscription_id }} -t <TARGET_TENANT_ID>
    Phase 3: Post-Transfer Actions 0
    • Import RBAC role assignments and custom roles
      Recreate role assignments using the exported file and principal mapping.
      Using Tenova CLI (recommended)
      tenova import-rbac -s {{ subscription_id }} -f rbac_assignments.json -m principal_mapping.json
      Or manually, per assignment
      # az role assignment create --assignee "<NEW_PRINCIPAL_ID>" \ # --role "<ROLE_NAME_OR_ID>" --scope "<SCOPE>"
    • Disable and re-enable system-assigned managed identities
      All system-assigned managed identities break after transfer. Toggle on each resource.
      Example: Reset identity on a VM
      # Disable then re-enable: az vm identity remove --ids "<VM_RESOURCE_ID>" --identities [system] az vm identity assign --ids "<VM_RESOURCE_ID>"
      Example: Reset identity on a Web App
      az webapp identity assign --name "<APP_NAME>" --resource-group "<RG>" \ --subscription {{ subscription_id }}
    • Rotate storage account access keys and SAS tokens
      Source tenant users may still have access via access keys. Rotate to revoke.
      List all storage accounts
      az storage account list --subscription {{ subscription_id }} --query "[].{name:name, rg:resourceGroup}" -o table
      Rotate keys for each account
      # Replace <ACCOUNT> and <RG>: az storage account keys renew --account-name "<ACCOUNT>" \ --resource-group "<RG>" --subscription {{ subscription_id }} --key key1
    • Reimport Azure Policy assignments
      Recreate policy assignments from the exported JSON.
      # For each assignment in policy_assignments.json: # az policy assignment create --name "<NAME>" \ # --policy "<POLICY_DEFINITION_ID>" \ # --scope "/subscriptions/{{ subscription_id }}"
    • Restore resource locks
      Recreate resource locks from the exported JSON.
      # For each lock in resource_locks.json: # az lock create --name "<LOCK_NAME>" \ # --lock-type "<CanNotDelete|ReadOnly>" \ # --resource-group "<RG>" --subscription {{ subscription_id }}
      Phase 4: Validation
      • Validate all services are working in the target tenant
        Test endpoints, verify database connectivity, check Key Vault access.
        Verify subscription tenant has changed
        az account show --subscription {{ subscription_id }} --query tenantId -o tsv
        List all resources to confirm access
        az resource list --subscription {{ subscription_id }} --query "length(@)" -o tsv
      • Run a follow-up Tenova scan in the target tenant
        Perform another scan to confirm all impacted resources have been remediated.
        tenova scan -s {{ subscription_id }}
      • Close maintenance window and notify stakeholders
        Confirm migration is complete and services are operational.
